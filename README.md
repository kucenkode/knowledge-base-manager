## Knowledge Base Manager

## Стек

- NestJS
- TypeScript
- In-memory storage

## Запуск

```bash
npm install
npm run start:dev
```

После запуска API доступен по адресу http://localhost:3000

---

## Концепция

Система моделирует цепочку зависимостей:

Document -> Audience -> VPC -> Page Sections

Документ влияет на аудиторию, аудитория содержит VPC, а поля VPC
используются при генерации секций страниц. Изменение одного узла отражается на зависимых сущностях.

---

## REST API

Методы:

GET `/documents` Список документов
POST `/documents` Создание документа
DELETE `/documents/:id` Удаление документа с каскадом
GET `/audiences` Список аудиторий и VPC
POST `/audiences` Создание аудитории
GET `/pages` Список страниц
POST `/pages` Создание страницы
GET `/impact/:docId` Предварительный расчет последствий каскада

---

## Каскадная логика

При удалении документа:

1.  Находятся все связанные аудитории через `docIds`
2.  Интервью этих аудиторий получают статус `outdated`
3.  Все VPC аудиторий получают статус `outdated`
4.  Находятся страницы, связанные с этими аудиториями
5.  Все связанные секции страниц получают статус `outdated`
6.  Возвращается полный Impact Report

Пример ответа:

```json
{
  "document": {
    "id": "d1",
    "name": "Продуктовый бриф"
  },
  "affectedAudiences": [
    {
      "id": "a1",
      "name": "B2B маркетологи"
    }
  ],
  "affectedVpcs": ["v1", "v2"],
  "affectedPages": [
    {
      "id": "p1",
      "name": "Главная",
      "sections": ["Hero", "CTA"]
    }
  ]
}
```

## Архитектурные решения

## ImpactService

Для расчета зависимостей выделен отдельный сервис.

Он отвечает за:

- поиск связанных сущностей;
- построение impact graph;
- формирование отчета;
- применение каскадных изменений.

Одна и та же логика используется для:

- DELETE `/documents/:id`
- GET `/impact/:docId`

---

## Хранилище

Используется in-memory, в production можно заменить storage слой без изменения бизнес-логики.

---

## Seed данные

Добавлены тестовые данные:

- 3 документа;
- 2 аудитории;
- 2 страницы;

---

## Статусы

Используются:

1. new - новая сущность
2. applied - актуальная сущность
3. outdated - устаревшая сущность (удаленная)

## Что можно улучшить

## Repository слой

Добавить:

- DocumentRepository
- AudienceRepository
- PageRepository

для разделения бизнес-логики и хранения и перейти на полноценную базу данных PostgreSQL.

## Ограничения

- данные хранятся только в памяти;
- после перезапуска сервера данные сбрасываются;

Для тестового задания основное внимание уделено архитектуре и
корректности каскадной логики.
